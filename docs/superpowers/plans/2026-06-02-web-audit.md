# Web / Browser-UI Threat-Modeled Audit — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (recommended for this plan — see note) or superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a threat-modeled security audit of the cancelchain browser/web UI — a report at `docs/superpowers/audits/2026-06-02-web-audit.md` plus strict-xfail demonstration tests in `tests/test_web_audit.py` — driven by a three-phase multi-agent Workflow fan-out.

**Architecture:** A Workflow fans out one analyst agent per adversary category over the in-scope code (discover), adversarially refutes each candidate (verify), and synthesizes survivors into a report-ready set. The controller writes the report and one `@pytest.mark.xfail(strict=True)` demonstration per finding, driven through the Flask test client. Ships as a docs PR; remediation of each finding is separate downstream work.

**Tech Stack:** Python 3.12+, pytest, the Flask **test client** (`test_client` fixture + `client.get(...)` patterns in `tests/test_browser.py`), `mill_block` for renderable data, `@pytest.mark.xfail(strict=True)`, the Workflow multi-agent tool.

> **Execution note — run inline, not subagent-driven.** The findings are synthesized by the Workflow (invoked by the controller, who holds the user's **explicit opt-in** — required to run a Workflow) and the report+tests are written directly from that in-context output. Recommend **inline execution (executing-plans)**: the controller runs Task 2's Workflow and writes Tasks 3–4 from its result. Tasks remain individually committable.

**Authoritative design:** `docs/superpowers/specs/2026-06-02-web-audit-design.md`. Read it first — the scope razor (esp. "public/unauthenticated is by-design" and "read-only/cookieless caps header findings at Low/Medium"), adversary categories, severity rubric.

## Task 2: Run the audit Workflow (discover → verify → synthesize)

> Requires the user's explicit Workflow opt-in. Mirror the prior five audits.

- [ ] **Step 1:** Author and run a Workflow with three phases:
  - **Discover:** six analyst agents (one per adversary category in the design's §"Adversary categories"), each handed the in-scope file list, the scope razor, and its lens. Each returns `{title, adversary, attack, code_path (file:line), precondition, impact, proposed_severity}` candidates + confirmed strengths.
  - **Verify:** ≥3 refuters per candidate try to disprove it against the trusted-boundary controls and the refuter ground rules below. Survives if a majority fail to refute.
  - **Synthesize:** dedupe, assign final severities, emit findings + strengths + a refuted-summary.
- [ ] **Step 2:** Persist to `/tmp/web-audit-findings.json` (scratch; not committed).

**Pre-seeded candidates the fan-out must confirm/refute** (illustrative, not exhaustive — walk every view and template):
- No HTTP security headers anywhere (CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, HSTS) — no `after_request`/`Talisman` in `application.py`. Likely the headline (Low/Medium, read-only/cookieless caps it).
- `return e` in every `browser.py` view (`:29-31,41-43,65-67,98-100`) — latent non-`HTTPException` return + info-disclosure depending on `DEBUG`/`PROPAGATE_EXCEPTIONS` (likely `DEBUG=False` in prod bounds it — confirm).
- XSS via `subject`/`human_subject` in `transaction.html:101-103` — confirm Jinja autoescape is on for `.html` and no `| safe`/`Markup` anywhere (likely refuted → strength).
- Malformed `integrity="…RI="crossorigin` (no space) on the jQuery `<script>` in `base.html:39` — SRI/CORS may not be enforced for the cross-origin asset (Low observation).
- Pagination `page`/`per_page` on `/chains` (`db.paginate`) — confirm Flask-SQLAlchemy's `max_per_page` bounds it (likely refuted/Low).

**Refuter ground rules (prevent false findings):**
- **"Public/unauthenticated browser views" is by design** (a block explorer) — NOT a finding.
- **Read-only + cookieless + login-less** caps header-hardening findings at Low/Medium (no session to steal, no action to forge) — do not inflate a missing header to High absent a concrete exploit.
- **Cross-reference, don't re-claim:** data validity → verification audit; `/api/*` auth → auth audit; query slowness → the perf roadmap.
- Credit real controls: Jinja autoescape (default on for `.html`); the `MillHashConverter`/`AddressConverter`/`SubjectConverter` constraining path params; ORM `filter_by` parameterization; SRI present on CDN assets; `DEBUG=False` default in production.

## Task 4: Write the demonstration tests

- [ ] Append one `@pytest.mark.xfail(strict=True)` test per finding to `tests/test_web_audit.py`, driven through the `test_client`:
  - **Header findings:** `client.get('/')` and assert the DESIRED header is present (e.g. `assert 'X-Content-Type-Options' in resp.headers`) under xfail; flips when the header is added.
  - **XSS findings (if any survive):** `mill_block` a transaction carrying an adversarial `subject`, GET `/transaction/<txid>`, assert the payload is escaped in `resp.data` (e.g. `b'<script>' not in resp.data`) — likely a *passing* regression documenting the autoescape strength rather than an xfail, if XSS is refuted.
  - **Error/disclosure findings:** assert the desired safe-error behavior.
- [ ] Run `uv run pytest tests/test_web_audit.py -q` — every xfail test must `xfail` (strict). A test that *passes* means the gap isn't real → drop or re-classify (or keep as a documented strength regression).
