> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (recommended for this plan — see note) or superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a threat-modeled security audit of the cancelchain CLI / operator surface — a report at `docs/superpowers/audits/2026-06-02-cli-audit.md` plus strict-xfail demonstration tests in `tests/test_cli_audit.py` — driven by a three-phase multi-agent Workflow fan-out.

**Architecture:** A Workflow fans out one analyst agent per adversary category over the in-scope code (discover), adversarially refutes each candidate (verify), and synthesizes survivors into a report-ready set. The controller writes the report and one `@pytest.mark.xfail(strict=True)` demonstration per finding (resource findings use a bounded-observation convention; all file I/O is `tmp_path`-confined). Ships as a docs PR; remediation of each finding is separate downstream work.

**Tech Stack:** Python 3.12+, pytest, Click `CliRunner` (the `runner` fixture + `runner.invoke(args=[...])` patterns in `tests/test_command.py`), `tmp_path`, `@pytest.mark.xfail(strict=True)`, the Workflow multi-agent tool.

> **Execution note — run inline, not subagent-driven.** The findings are synthesized by the Workflow (invoked by the controller, who holds the user's **explicit opt-in** — required to run a Workflow) and the report+tests are written directly from that in-context output. Recommend **inline execution (executing-plans)**: the controller runs Task 2's Workflow and writes Tasks 3–4 from its result. Tasks remain individually committable.

**Authoritative design:** `docs/superpowers/specs/2026-06-02-cli-audit-design.md`. Read it first — the scope razor (esp. the operator-trust model and the four cross-referenced audits), adversary categories, severity rubric, and the bounded-observation test convention live there.

## Task 2: Run the audit Workflow (discover → verify → synthesize)

> Requires the user's explicit Workflow opt-in. Mirror the prior four audits.

- [ ] **Step 1:** Author and run a Workflow with three phases:
  - **Discover:** six analyst agents (one per adversary category in the design's §"Adversary categories"), each handed the in-scope file list, the scope razor, and its lens. Each returns `{title, adversary, attack, code_path (file:line), precondition, impact, proposed_severity}` candidates + confirmed strengths.
  - **Verify:** ≥3 refuters per candidate try to disprove it against the trusted-boundary controls and the refuter ground rules below. Survives if a majority fail to refute.
  - **Synthesize:** dedupe, assign final severities, emit findings + strengths + a refuted-summary.
- [ ] **Step 2:** Persist to `/tmp/cli-audit-findings.json` (scratch; not committed).

**Pre-seeded candidates the fan-out must confirm/refute** (illustrative, not exhaustive — walk *every* command body, incl. `txn/*` and `wallet create`):
- **`wallet create` writes the private-key PEM world-readable + unencrypted + silent-overwrite** — `Wallet.to_file` → `open(filename, 'wb')` (`wallet.py:~225`, called from `command.py:913`) has no `chmod 0o600`, no `O_EXCL`, and `wallet create` passes no passphrase. A different local user/process can read the key → external adversary, NOT self-harm (see refuter rules). Likely the strongest CLI finding; the crypto audit did not claim file permissions.
- `import` has no line-length / file-size bound (`command.py:478,480-482`); a single huge line is read whole.
- `import` dedup gate keys on the self-reported `block.block_hash` field (`command.py:483`) and advances the progress bar whether the block is added or skipped (`command.py:485`) — a crafted colliding hash silently drops a block with no warning. (Add-under-wrong-identity is ruled out: `add_block` revalidates; trace to confirm.)
- `export` append-vs-overwrite decided by `read_last_line` + self-reported-hash chain match (`command.py:435-438`) against a possibly pre-seeded file; path/overwrite/symlink handling on `click.Path()` (no `exists`/dir guard).
- `host_api_client` selects the signing wallet from the host URL username and sends to that host (`command.py:81-105`) — credential-selection confusion / SSRF-shaped signature leak via a hostile `--host`/`CC_DEFAULT_COMMAND_HOST`.
- `txn/*` (`transfer`/`subject`/`forgive`/`support`) `--host`/`--wallet-file`/`--txn-wallet` routing — Category-3/4 surfaces structurally identical to the seeded commands.
- `console.print_exception()` / broad `except` / bare `raise Exception` (`command.py:115,461,487`) leaking key paths/signatures/secrets, or masking partial failure.

**Refuter ground rules (prevent false findings):**
- **Operator-trust model:** the operator running the CLI is trusted. "The operator can crash/clobber their own node/files" is **self-harm, not a finding** — UNLESS an external party realistically supplies the triggering artifact (a shared `.jsonl`, a config value, a dropped `*.pem`).
- **Do NOT kill the `wallet create` permissions finding as self-harm.** Writing a credential world-readable/unencrypted exposes it to a *different* local user/process (an external adversary). The operator runs the command, but the harm accrues to a third party reading the file — that is a genuine finding, graded on the exposure.
- **Cross-reference, don't re-claim:** block validity → verification audit (import routes through `node.add_block` → `chain.add_block` → `Chain.validate_block`); key parsing → crypto audit; network/peer exhaustion → networking audit; auth → auth audit. A candidate that reduces to one of these is a cross-reference, not a new CLI finding.
- Credit real controls: `click.Path(exists=True)` on the `import`/wallet-file/txn inputs; the host-username→local-wallet self-binding (`host_address` + `wallets.get(address)`); the validated `add_block` path.

## Task 4: Write the demonstration tests

- [ ] Append one `@pytest.mark.xfail(strict=True)` test per finding to `tests/test_cli_audit.py`, driven through the `runner` fixture with `tmp_path` files:
  - **Correctness findings:** assert the DESIRED post-fix behavior (e.g. `import` of a crafted file rejects / bounds it; `host_api_client` refuses a credential-selection trap) under xfail; flips to a passing regression on remediation.
  - **Resource findings:** bounded-observation — the demonstration asserts the DESIRED post-fix *rejection* (e.g. an oversized single line raises a size-limit error), which fails today because no guard exists → valid strict-xfail, flipping to a passing regression when the bound lands. Drive only to a *small, safe* bound; **no test exhausts real memory/disk.** (A test that merely reads a large-but-safe input and succeeds does NOT demonstrate the finding — invert it to assert rejection.)
- [ ] Run `uv run pytest tests/test_cli_audit.py -q` — every test must `xfail` (strict). A test that *passes* means the gap isn't real → drop or re-classify.
