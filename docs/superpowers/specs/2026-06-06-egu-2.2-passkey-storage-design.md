# EGU #2.2 — passkey-anchored wallet storage — design

**Date:** 2026-06-06
**Status:** Implemented (#172)
**Issue:** #172 (sub-project 2 of EGU #2 / #152)
**Type:** New client module (vanilla JS / Web Crypto + WebAuthn + IndexedDB) — additive; no change to the Python chain.

## Summary

Persist the browser wallet's private key **encrypted at rest in IndexedDB**, with
the at-rest encryption key derived from the **WebAuthn PRF extension** of a
passkey. Unlock = passkey touch → PRF output → decrypt → reconstruct the `Wallet`
from #2.1. The unencrypted private key never persists; the symmetric key is
re-derived from the authenticator at each use and never stored.

Builds on the merged signing core (#170/#171): `Wallet.exportPrivateKeyB58()` /
`Wallet.fromPrivateKeyB58()` are the persistence seam — no change to `gc-wallet`.

Scope: **require PRF** (fail clearly if unsupported), **single wallet per
device**. Backup/recovery, message-signing UX, packaging, and any non-PRF
fallback are later sub-projects.

**Authenticator support & unlock UX.** PRF is keyed by the credential, not the
gesture, so **synced/software passkeys work** (iCloud Keychain, Google Password
Manager, Bitwarden — recent versions; the requirement is PRF capability, not a
hardware key). WebAuthn has no silent assertion, so unlock is always **one
explicit step**, but how heavy is tunable via `userVerification` (below): with an
already-unlocked provider and the default, it can be a single confirm click — no
fresh biometric each time. It is low-friction, not fully touchless.

## Architecture — dependency injection (so the crypto logic is Node-testable)

WebAuthn and IndexedDB are browser-only and cannot run in the zero-npm
`node --test` CI path. So the orchestration depends on two **injected
interfaces**, not on `navigator.credentials` / `indexedDB` directly. The
security-critical logic (encryption envelope + enroll/unlock orchestration) is
then fully Node-testable with fakes; only two thin real adapters are
browser-only and manually verified.

- **`passkey` interface:**
  - `isSupported() -> Promise<boolean>` — WebAuthn + PRF available.
  - `enroll(opts) -> Promise<{ credentialId: string, prfOutput: Uint8Array }>` —
    create a passkey (PRF enabled) and return its id + the 32-byte PRF output.
  - `unlock(credentialId) -> Promise<Uint8Array>` — assert the passkey, return
    the 32-byte PRF output.
- **`store` interface (single record):**
  - `get() -> Promise<record | null>`, `put(record) -> Promise<void>`,
    `delete() -> Promise<void>`.

## Components (`clients/wallet/`)

- **`gc-envelope.mjs`** — pure crypto, Node-tested. `seal(prfOutput, bytes)` and
  `open(prfOutput, { iv, ciphertext })`. No WebAuthn/IndexedDB knowledge.
- **`gc-store.mjs`** — orchestration, Node-tested with fakes:
  `enroll(wallet, { passkey, store }, opts)`, `unlock({ passkey, store })`,
  `hasWallet(store)`, `clear(store)`. Wires the envelope + injected interfaces.
- **`gc-passkey-webauthn.mjs`** — real `passkey` over `navigator.credentials` +
  the PRF extension. Browser-only; manually verified.
- **`gc-store-idb.mjs`** — real `store` over IndexedDB (one DB, one record).
  Browser-only; manually verified.
- **`passkey-wallet-demo.html`** — a manual-test page to exercise real
  enroll/unlock on a PRF-capable device.

Dependency order: `gc-store` → `gc-envelope` (+ injected `passkey`/`store`); the
two adapters implement the interfaces. `gc-envelope` reuses `gc-crypto.mjs`
helpers (base64) where useful.

## Cryptographic envelope (the security-critical core)

All via Web Crypto (`crypto.subtle`); browser + Node identical.

- **Key derivation:** `prfOutput` (32 bytes from the authenticator) →
  `importKey('raw', prfOutput, 'HKDF')` → `deriveKey({ name: 'HKDF', hash:
  'SHA-256', salt: new Uint8Array(0), info: utf8('gc-wallet-aesgcm-v1') }, …,
  { name: 'AES-GCM', length: 256 }, …, ['encrypt','decrypt'])`. An empty HKDF
  salt is sound here (RFC 5869) because the PRF output is already high-entropy
  keying material; the versioned `info` label is the domain separator and the
  PRF output is never used directly as an AES key.
- **Encryption:** AES-GCM-256, a **fresh random 12-byte IV per `seal`**
  (`crypto.getRandomValues`). `seal` returns `{ iv, ciphertext }` (ciphertext
  includes GCM's 16-byte auth tag, appended by Web Crypto). `open` decrypts and
  **fails closed** if the tag doesn't verify (wrong authenticator / tampered
  store) — never returns partial/garbage plaintext.
- **Plaintext:** the wallet's private key in its b58 form
  (`utf8(wallet.exportPrivateKeyB58())`). On `open`, decode UTF-8 →
  `Wallet.fromPrivateKeyB58()`. (Encrypting the b58 string avoids adding a raw
  PKCS8 accessor to `gc-wallet`; the bytes are equivalent.)

## WebAuthn PRF flow (the real adapter)

- **PRF salt:** a fixed app constant (the PRF "first" input). It is **not
  secret** — it's a domain label; the same salt at enroll and unlock yields the
  same PRF output for a given credential. Define it once (e.g.
  `utf8('gc-wallet-prf-v1')`).
- **Enroll:** `navigator.credentials.create({ publicKey: { …, extensions: {
  prf: { eval: { first: SALT } } } } })`. Prefer the create-time PRF result if
  the platform returns it (`getClientExtensionResults().prf.results.first`); if
  absent, immediately do one `get()` with the same `prf.eval.first` to obtain the
  output. Return `{ credentialId: base64url(rawId), prfOutput }`.
- **Unlock:** `navigator.credentials.get({ publicKey: { allowCredentials:
  [{ id: <credentialId> , type: 'public-key' }], extensions: { prf: { eval: {
  first: SALT } } } } })` → `getClientExtensionResults().prf.results.first` (32
  bytes). If `prf.results` is absent (PRF unsupported), throw `UnsupportedError`.
- **`isSupported()`** probes WebAuthn availability (`window.PublicKeyCredential`)
  and, where determinable, PRF; the authoritative check is whether `prf.results`
  comes back on enroll — surface `UnsupportedError` cleanly if not.
- **`userVerification`:** caller-configurable via `opts`, **default
  `'preferred'`** — uses UV when the authenticator offers it for free (an
  already-unlocked Bitwarden vault, a platform biometric) but does not force a
  heavier prompt or hard-fail when it isn't, giving the lightest unlock that
  still benefits from UV when available. Both enroll (`authenticatorSelection
  .userVerification`) and unlock (`get`'s `userVerification`) use it. An
  embedding app may raise it to `'required'` for higher-value contexts.
- RP config (`rpId`, `rpName`, user id/name) is caller-provided via `opts` — the
  module is embeddable, nothing hardcoded.

## Stored record (IndexedDB, single row)

`{ version: 1, address, credentialId, iv, ciphertext }` — all binary fields
base64-encoded for JSON-friendly storage. `address` is kept for display/UX
without unlocking. No secret is stored in plaintext.

## Data flow

- **Enroll:** `isSupported()` gate → `passkey.enroll(opts)` →
  `seal(prfOutput, utf8(wallet.exportPrivateKeyB58()))` → `store.put({version,
  address, credentialId, iv, ciphertext})`. Returns the address.
- **Unlock:** `store.get()` (null → `NoWalletError`) → `passkey.unlock(
  credentialId)` → `open(prfOutput, record)` → `Wallet.fromPrivateKeyB58(...)`.

## Error handling (typed errors)

- `UnsupportedError` — WebAuthn/PRF unavailable (checked up front + on enroll,
  and thrown by the real adapter when an assertion returns no PRF result). Typed
  errors live in `gc-errors.mjs`, shared by `gc-store` and the adapters.
- `NoWalletError` — `unlock` with nothing stored. (`clear` is **idempotent** — a
  no-op when empty, not an error; "ensure no wallet stored" is the contract.)
- Unknown record `version` — `unlock` fails fast with a clear error rather than
  mis-decoding fields (the seam where a future schema migration plugs in).
- WebAuthn user-cancel/abort or a `null` assertion — surfaced as a clear error
  (DOMException propagated / explicit throw), never a `TypeError`.
- Decrypt failure — `open` throws (GCM tag mismatch); never returns garbage.
No silent fallback anywhere.

## Security properties (explicitly, so they're checkable)

1. **At rest, only ciphertext + non-secrets persist.** IndexedDB holds the
   AES-GCM ciphertext, IV, credentialId, and address — never the private key or
   the AES key.
2. **The AES key is ephemeral and authenticator-bound.** It is HKDF-derived from
   the PRF output at the moment of use and discarded; it is never stored and
   cannot be produced without a passkey touch on the enrolled authenticator.
3. **Tamper-evident.** AES-GCM authentication makes any modification of the
   stored ciphertext/IV fail `open` closed.
4. **Domain-separated, versioned.** HKDF `info` and the PRF salt are
   app-specific and versioned (`…-v1`), so the key schedule can evolve without
   ambiguity.
5. **No key exfiltration path widened.** The module never exposes the raw AES key
   or PRF output to callers; `unlock` returns a `Wallet` (whose private key stays
   a non-extractable-by-default concern of #2.1's `Wallet`).

## Testing

- **Node (`node --test`, zero npm):**
  - `gc-envelope`: `seal`→`open` round-trip with a fixed fake 32-byte PRF output;
    tamper rejection (flip a ciphertext/IV byte → `open` throws); distinct IVs
    across seals; wrong-key `open` fails.
  - `gc-store`: full **enroll → unlock** round-trip using a `Map`-backed fake
    `store` and a fake `passkey` returning a fixed PRF output → the recovered
    `Wallet` signs identically to the original (cross-check against #2.1).
    Plus `hasWallet`, `clear`, `NoWalletError` on empty unlock, and that a
    *different* fake PRF output fails `open` (simulating a wrong authenticator).
- **Browser (manual, documented):** `passkey-wallet-demo.html` exercises the real
  `gc-passkey-webauthn` + `gc-store-idb` on a PRF-capable device: enroll, reload,
  unlock, sign; confirm IndexedDB holds only ciphertext. These adapters are thin
  glue with no branching logic.

## Decisions log

- **Dependency injection** (injected `passkey`/`store`) — makes the envelope +
  orchestration Node-testable; keeps the two real browser adapters thin.
- **Require PRF, single wallet** — smallest clean scope; pure security model
  (key never leaves the authenticator); multi-wallet via browser profiles.
- **`userVerification` default `'preferred'`, caller-configurable** — lightest
  unlock matching the existing 2b2f passkey feel (synced passkeys like Bitwarden
  supported; one low-friction step, not touchless), while embedders can raise to
  `'required'`.
- **HKDF→AES-GCM-256, random IV, GCM auth** — standard authenticated encryption;
  domain-separated/versioned key schedule.
- **Encrypt the b58 private key** — reuses #2.1's seam, no `gc-wallet` change.
- **Zero npm preserved** — Node tests for envelope+orchestration; manual browser
  verification for the adapters.
