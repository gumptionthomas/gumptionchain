# Contributing to GumptionChain

Thanks for your interest in improving GumptionChain. This document covers how
to set up a development environment, the quality gates your change must pass,
and the workflow for getting it merged.

By participating in this project you agree to abide by the
[Code of Conduct](CODE_OF_CONDUCT.md).

## Development environment

GumptionChain uses [uv](https://docs.astral.sh/uv/) for environment and
dependency management. `uv.lock` is committed and authoritative for
reproducible builds.

```console
$ git clone https://github.com/gumptionthomas/gumptionchain.git
$ cd gumptionchain
$ uv sync --group dev
$ uv run pre-commit install
```

## Quality gates

Every change must pass the same checks CI enforces. Run them locally before
pushing:

```console
$ uv run ruff check src tests          # lint
$ uv run ruff format --check src tests  # formatting
$ uv run mypy                           # type checking (strict)
$ uv run pytest                         # full test suite
$ uv run gumptionchain db check         # schema matches migration history
```

- **Tests** run against a per-test temporary SQLite database. Multiprocessing
  tests are skipped unless you pass `--runmulti`.
- **Coverage:** `uv run pytest --cov=gumptionchain`.
- **Style:** Python ≥ 3.12, 80-column lines, single quotes. `ruff` and `mypy`
  configuration lives in `pyproject.toml`.

## Schema migrations

The schema is managed with Alembic via Flask-Migrate. After editing a model,
generate and **hand-review** a migration:

```console
$ uv run gumptionchain db migrate -m "describe the change"
$ uv run gumptionchain db upgrade
```

Autogenerated migrations are a starting point, not a finished product —
CHECK constraints, server defaults, and type-flavor distinctions often need
manual edits. `gumptionchain db check` is a CI gate that enforces the
`db.create_all()` schema matches the migration history.

## Workflow

1. **Branch.** Every change goes through a branch and pull request — including
   refactors, one-line fixes, and docs. Name branches `<type>/<short-desc>`
   (e.g. `feat/peer-gossip-retry`, `fix/signing-key-load-race`,
   `docs/api-reference`).
2. **Commit** using [Conventional Commits](https://www.conventionalcommits.org/)
   (`feat(scope): …`, `fix: …`, `refactor: …`, `docs: …`).
3. **Keep PRs focused.** A fix PR fixes the thing it was opened for; adjacent
   refactors become their own PRs. Bundling expands blast radius and
   complicates revert.
4. **Open a pull request** against `main` with a summary of what changed and
   how to test it. Make sure CI is green.
5. **Squash merge** once approved (`--squash`), and delete the branch.

## Dependencies and supply chain

- Add or bump runtime dependencies in `pyproject.toml`'s `[project]`, dev
  dependencies under `[dependency-groups].dev`, then run `uv lock` and commit
  the lockfile change.
- Remediate CVEs with `uv lock --upgrade-package <name>`, never by editing
  pins in isolation. CVE scanning (pip-audit) runs in CI on every PR.

## Reporting bugs and requesting features

Use the issue templates under
[`.github/ISSUE_TEMPLATE`](.github/ISSUE_TEMPLATE). For security
vulnerabilities, **do not open a public issue** — follow
[SECURITY.md](SECURITY.md) instead.
